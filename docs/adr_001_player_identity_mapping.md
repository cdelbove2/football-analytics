# ADR 001: Player identity mapping table

## Status
Accepted

## Context
Each data source (FBref, Transfermarkt, Understat, etc.) uses its own
internal identifier for the same real-world player. To combine
performance stats with market value and other cross-source data, we need
a way to say "these IDs refer to the same person."

## Decision
Maintain a single table, `core.player`, with:
- An internal `player_id` (UUID), generated by us.
- One nullable `*_id` column per source (`fbref_id`, `transfermarkt_id`, ...).
- A `match_status` field tracking how confident we are in the mapping:
  `unmatched`, `auto_matched`, `confirmed`, `needs_review`.
- A `match_confidence` score for fuzzy matches.

New sources are added as new nullable columns - no restructuring of
existing data required.

## Matching pipeline (planned)
1. **Exact match**: name (normalised) + nationality + date of birth match
   exactly across sources -> `auto_matched`, confidence = null (treated as 1.0).
2. **Fuzzy match**: name similarity above a threshold (e.g. using
   `thefuzz`/rapidfuzz) + matching club/nationality -> `needs_review` with
   a `match_confidence` score.
3. **Manual review**: anything below the fuzzy threshold, or any
   `needs_review` row, gets reviewed by hand (initially via a simple
   script/notebook, later via a small review UI) and flipped to
   `confirmed` or split into separate `core.player` rows if it was a
   false match.

## Consequences
- Downstream dbt models join on `player_id`, never directly on a source's
  native ID, keeping the rest of the pipeline source-agnostic.
- `match_status` lets us filter out low-confidence matches from
  user-facing features until reviewed, without deleting the candidate
  mapping.
